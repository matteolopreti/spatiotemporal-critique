### ISSUES

*   **[Severity: High] Misleading "no migration steps" claim for self-hosted upgrades** — Stating "requires no migration steps" while simultaneously recommending a database backup is contradictory and dangerous. Self-hosted upgrades likely execute automated database schema migrations on startup. If an admin assumes "no migrations" means "no database changes," they may run the upgrade outside of a maintenance window or skip critical pre-migration checks.
    *   *Concrete Fix:* Revise to: *"Meridian 2.4 requires no manual migration steps. Database schema updates will run automatically upon startup; as always, we recommend a full database backup before upgrading."*
*   **[Severity: Medium] Unspecified authorization scopes for new API endpoints** — The notes introduce `/activity` and `/reviews` as read-only but do not specify their access controls. Because `/activity` contains sensitive audit logs (sign-ins, exports), exposing it without clarifying that it requires admin-level tokens is a security communication gap.
    *   *Concrete Fix:* Revise to: *"the REST API adds two read-only endpoints: `/activity` (restricted to workspace administrators) and `/reviews` (respecting standard document access permissions)."*
*   **[Severity: Medium] Ambiguity in "untouched sections" for partial approvals** — The phrase *"a returned draft keeps approvals for untouched sections"* implies a highly complex document state. It is unclear how the system defines "untouched" (e.g., character-level diffs, block-level IDs, or template fields) and whether editing a dependent section invalidates prior approvals.
    *   *Concrete Fix:* Clarify the mechanism: *"Templates can mark distinct content blocks as review-required, so partial approvals are possible: a returned draft preserves approvals for unmodified template blocks, requiring re-review only for edited blocks."*
*   **[Severity: Low] Undefined fallback for seat reassignment** — The text states *"ownership transfers to a nominated successor"* but does not clarify if nominating a successor is mandatory or what happens if one is not selected during deprovisioning.
    *   *Concrete Fix:* Revise to: *"Seats can now be reassigned without deleting the departing user's documents; admins are prompted to transfer ownership to a nominated successor, preserving the full edit history."*
