### PRESERVE

The following architectural choices are highly robust, represent industry best practices for high-stakes migrations, and must be preserved in subsequent revisions:

*   **The Strangler Facade Pattern:** Introducing the billing facade inside the monolith boundary (*Phase A*) decouples the routing logic from the physical location of the services. This allows for fine-grained, low-risk traffic shifting without requiring coordinated deployments of both the monolith and the target services.
*   **Shadow Validation Cycles:** Requiring two full billing cycles of shadow comparison for invoices (*Phase B*) and two accounting periods of parallel ledger reconciliation (*Phase D*) ensures that subtle edge cases (such as rounding errors, proration differences, and timezone mismatches) are caught before any customer-facing impact occurs.
*   **Immutable Ledger Rollback Strategy:** The decision *not* to delete service-created ledger entries during a Phase D rollback, but instead to mark the projection as inactive and reconcile the stream, is excellent. This preserves the audit trail and prevents dangerous database state mutations that would alarm financial auditors.
*   **Idempotent Migration Scripts:** Designing the data migration scripts to write source identifiers (source system, table, primary key) alongside migrated rows is the correct way to ensure traceability and allow for safe retries.

### ISSUES

#### [Severity: HIGH] Circular Dependency Deadlock between Phase B and Phase D
*   **The Problem:** The migration plan contains a logical deadlock in its execution sequence. *Phase B* states: *"Phase B cannot start until the canonical customer-balance event schema produced by Phase D has been accepted by finance operations and implemented by downstream consumers."* However, the *Cutover Checklist* states: *"Confirm Phase D starts only after Phase B invoice extraction has completed production traffic promotion..."* Under these rules, neither phase can proceed.
*   **The Concrete Fix:** Decouple the *schema definition* from the *service implementation*. Update Phase A to include the collaborative design, financial sign-off, and downstream implementation of the canonical customer-balance event schema. This allows Phase B to emit mock-conforming events and Phase D to build its ingestion pipeline independently, breaking the deadlock.

#### [Severity: HIGH] Idempotency Contradiction in Cutover Checklist
*   **The Problem:** There is a direct technical contradiction between the Data Migration section and the Cutover Checklist. The Data Migration section states the script is *"idempotent and safely re-runnable."* However, the Cutover Checklist states: *"...inserting transformed records into each target service table, preserving source IDs for traceability but appending new rows for every source record processed during the batch."* If the script blindly appends new rows on every execution, it is not idempotent and will duplicate financial records (invoices, payments, ledger entries) if a batch is interrupted and retried.
*   **The Concrete Fix:** Revise the Cutover Checklist to specify an `UPSERT` (insert-or-update) behavior. The script must check for the existence of the preserved `source_system` and `source_primary_key` in the target database. If they exist, it must update the record or skip it, rather than appending a new row.

#### [Severity: MEDIUM] Phase C Rollback Refund/Capture Routing Gap
*   **The Problem (Hypothesis):** If Phase C (Payments) is rolled back to the monolith after a cohort has run live on the Payment Service, some payments will have been authorized and captured by the new Payment Service. If a customer subsequently requests a refund or if an in-flight authorization needs to be captured *after* rollback, the monolith's legacy gateway adapter may lack the transaction tokens, state history, or API keys required to interact with those specific transactions.
*   **The Concrete Fix:** Explicitly state in the Phase C rollback plan that the billing facade will route post-payment actions (refunds, voids, captures) based on the *originating service ID* of the transaction, ensuring that the Payment Service can still process lifecycle events for transactions it created even if new payment checkouts have reverted to the monolith.

#### [Severity: MEDIUM] Active Cohort Migration Race Condition
*   **The Problem:** The plan relies on running a historical migration job, freezing mutations, draining queues, running a delta sync, and switching flags. If the historical migration of a cohort takes a significant amount of time, the volume of delta writes generated by active users in that cohort during the migration window could be massive, leading to extended downtime during the "freeze" or write conflicts during the delta sync.
*   **The Concrete Fix:** Introduce a dual-write or Change Data Capture (CDC) replication phase for the active cohort *prior* to the freeze. The historical migration runs first, CDC keeps the target service database in near-real-time sync, and the freeze window is used solely to drain queues and verify the final checksums, reducing the freeze duration to seconds.

#### [Severity: LOW] Ambiguity of Phase C "Shadow Mode"
*   **The Problem:** Phase C claims to run the Payment Service in "shadow mode" by simulating processor calls using recorded gateway responses. In distributed systems, "shadowing" typically implies processing duplicate live traffic. Because credit cards cannot be charged twice, this is actually a synthetic replay/simulation. Using the term "shadow mode" here creates false expectations for SREs and business stakeholders regarding the use of live production traffic.
*   **The Concrete Fix:** Rename Phase C's initial phase from "shadow mode" to "Simulation and Replay Mode" to clarify that no live external gateway calls are being duplicated.

#### [Severity: LOW] Missing Transactional Outbox Pattern for Domain Events
*   **The Problem (Hypothesis):** The plan states that services will *"publish domain events through the existing message bus."* If a service writes to its database but the network call to the message bus fails (or vice versa), the system will fall out of sync, violating the core success criterion of a "clean audit trail."
*   **The Concrete Fix:** Specify that all services must use the Transactional Outbox pattern (writing events to an `outbox` table in the same database transaction as the business state change, then publishing them asynchronously) to guarantee at-least-once delivery of domain events.

#### Status: CONTESTED (Requires Revision)

While the operational safety guardrails, rollback thresholds, and financial audit principles in this plan are exceptionally well-thought-out, the document cannot be approved in its current form due to the **Phase B/D circular dependency deadlock** and the **idempotency contradiction** in the cutover checklist. 

The final decision to proceed belongs to the human owner. Address the high and medium severity issues listed above to transition this plan to a corroboratable, production-ready state.
